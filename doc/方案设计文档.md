## 一、项目概述：做了什么、解决了什么问题

### 1.1 一句话定位

MedicineBoxNotes 是一款**面向家庭场景的 iOS 健康记录 App**，把"家庭成员的病历 / 处方 / 检查附件 / 家中常备药"集中到一处，并以**离线优先、本地可追溯、端侧 AI 辅助整理**为核心设计原则。

### 1.2 它要解决的真实问题

| 痛点 | App 的应对 |
|---|---|
| **医疗信息分散在多个纸质单据和微信里**：病历、处方、检查单、药盒说明书全是纸，搬家/复诊时翻不到 | 用相机把所有单据电子化，自动 OCR + AI 结构化，按家庭成员归档，支持全文检索 |
| **看病时医生问"之前吃过什么药、看过什么病"，答不上来** | 病历按时间倒序、按年份分组，附件可搜索；处方与药箱联动可溯源 |
| **家里常备药吃完了才发现、过期了还在吃** | 药箱管理库存 + 低库存预警（≤5）+ 服药计划派生今日待办 |
| **复诊总是忘** | 设置复诊日期后，本地通知在**前一天 9:00 + 当天 9:00** 两次提醒，点通知直达病历 |
| **想问"女儿去年咳嗽吃过什么药"这种问题，只能手动翻** | 底部"查询"Tab：基于本地病历/附件 OCR·AI 文本/药箱数据做检索召回 + 端侧 Gemma 生成**带引用来源**的 grounded answer |
| **医疗隐私敏感，不敢把数据上传云端做 AI** | 默认本地存储；AI 推理**完全在设备端**（Gemma 4 E2B + LiteRT-LM）；唯一的联网（DuckDuckGo）只发用户问题、不传任何病历/药品数据 |
| **AI 会编造（幻觉），用在医疗上很危险** | 端到端的防幻觉设计：提示词硬规则 + 输出 JSON 全可选 + "没有/不知道"归一化 + 问答安全门（无引用必须承认无知）+ 全程风险确认 + 结果标记"待核对" |

### 1.3 产品形态与核心价值

- **形态**：iOS 原生 App（iOS 17+），底部 5 Tab：首页 / 病历 / 药箱 / 查询 / 设置。
- **三大资产**：病历档案（MedicalRecord）、家庭药箱（MedicineItem）、家庭成员（FamilyMember），以及派生的服药日志（MedicationLog）。
- **两条主线**：
  1. **录入线**：拍照 → Vision OCR → 端侧 AI 结构化抽取 → 用户确认 → 入库。
  2. **使用线**：首页今日待办 / 复诊提醒 / 低库存 / 自然语言查询。
- **核心价值主张**：把零散的家庭医疗纸面信息变成**可搜索、可追溯、可提醒、可在本地问答**的结构化资产，同时把数据和 AI 都锁在本机。

### 1.4 目标用户与典型场景

- **用户**：承担"家庭健康记账人"角色的人（通常是中年子女），同时管理自己和父母/孩子的就医与用药。
- **典型场景**：
  - 带父母看病，医生问"上次心内科什么时候、开了什么药"——打开病历 Tab 一秒调出。
  - 整理一堆药盒：拍正/反/侧面 + 说明书，AI 合并 OCR 后整理出名称/剂量/频率/库存。
  - 孩子半夜咳嗽，想确认之前过敏开的什么药——查询 Tab 自然语言提问，答案带来源病历。
  - 复诊前一天收到提醒，点通知直接进到对应病历。

## 二、整体方案：架构与技术栈

### 2.1 技术栈

| 层 | 选型 |
|---|---|
| 平台 | iOS 17+ |
| 语言 | Swift 5 |
| UI | SwiftUI（自定义浮动 TabBar、纸感卡片体系） |
| 数据 | SwiftData（`@Model`），兼容 CloudKit 自动同步 |
| OCR | Apple Vision Framework（`VNRecognizeTextRequest`，中英双语） |
| 图片 | UIKit（`UIImagePickerController` / `UIActivityViewController`），ImageIO 下采样 |
| 通知 | `UserNotifications`（`UNCalendarNotificationTrigger`） |

| 外部搜索 | DuckDuckGo Instant Answer API（仅查询页手动开启，5s 超时） |
| 模型分发 | 运行时下载（Hugging Face / HF-Mirror，断点续传） |

### 2.3 模块划分（当前正在进行的模块化拆分）

- `MedicineBoxNotesApp.swift`：App 入口 + SwiftData 全部模型 + 根导航 + 病历/药箱/查询/设置主页面 + OCR/PDF/通知/图片存储内嵌服务（**仍是单文件大文件，约 4900 行，后续持续拆分**）。
- `MedicalAI*` + `GemmaEngine`：端侧 AI 抽象层与推理引擎。
- `Services/`：模型下载、AI 配置、联网搜索、今日计划、图片预处理。
- `Views/`：HomeView、Settings 下三个页面。
- `Components/`：纸感组件库（PaperCard、FloatingTabBar、MemberDot 等 15 个）。
- `Theme/`：设计 token（颜色/字体/度量/成员色板）。
- `medicine_box_design/`：设计原型（HTML/JSX，steady 与 bold 两套方案）。

## 三、数据模型与存储设计

### 3.1 SwiftData 模型清单

> 全部模型为兼容 CloudKit，**不使用 `@Attribute(.unique)`**，唯一性靠 `UUID v4` 默认值；关系用 SwiftData 默认 `.nullify`，无显式 cascade 注解。

#### 3.1.1 枚举（持久化为 raw String）

- `AttachmentType`：病历 / 处方 / 检查单 / 其他
- `MedicineScanType`：药盒正面 / 药盒反面 / 药盒侧面 / 说明书 / 其他
- `StorageBackend`：`local`（仅本机）/ `iCloud`（跨 Apple 设备同步）

#### 3.1.2 实体

| 实体 | 职责 | 关键字段 |
|---|---|---|
| **FamilyMember** | 家庭成员 | `id/name/relationship/emoji/createdAt`，反向 `records: [MedicalRecord]` |
| **MedicalRecord** | 病历 | `visitDate/hospitalName/department/doctorName/chiefComplaint/diagnosis/doctorAdvice/followUpNote/followUpDate?`，关系 `member?`、`attachments[]`、`prescriptions[]` |
| **RecordAttachment** | 病历附件照片 | `imageData(.externalStorage)/ocrText/aiSummary/aiStructuredJSON/aiSearchableText/aiProviderRaw/attachmentType` |
| **Prescription** | 处方药品（隶属于病历） | `medicineName/dosage/frequency/duration(Int)/note`，反向 `record?` |
| **MedicineItem** | 药箱药品 + 服药计划 + 来源溯源 | `name/dosage/frequency/durationDays/note/aiSummary/aggregatedOCRText/stock/sourceRecordID?/sourceHospital/sourceVisitDate?/sourceMemberName/sourcePrescriptionID?` + 服药计划 `scheduledTimes[]/planStartDate?/planEndDate?/isActive`，反向 `scans[]` |
| **MedicationLog** | 服药打卡日志 | `medicineID(UUID 裸)/memberName(裸字符串)/scheduledTime/scheduledDate/takenAt` |
| **MedicineScanAsset** | 药盒扫描图 | `imageData(.externalStorage)/ocrText/scanType`，反向 `medicine?` |

#### 3.1.3 关键设计：病历 ↔ 药箱的"软锚定"

- 药箱条目通过 `sourcePrescriptionID`（指向 `Prescription.id`）与处方**软关联**，而非 SwiftData 强关系。
- **删除病历或取消同步处方时，只解开引用，绝不删除药箱条目**——因为药箱的 `stock`、`scheduledTimes`、计划是用户资产。
- `MedicationLog` 用裸 `medicineID` UUID + 裸 `memberName` 字符串关联（不走 SwiftData 关系），因此**删药品、删成员、改名时必须手动 fetch 并清理孤立记录**（见 §6.3）。

### 3.2 存储策略：三级回退 + 两套容器

`StorageCoordinator` 在启动时按 **iCloud → 本地 → 内存** 三级回退构建 `ModelContainer`：

1. 读 `UserDefaults["storage.backend"]` → 尝试构建对应容器。
2. iCloud 构建失败 → 回退本地，写 `lastError` 文案。
3. 本地也失败 → 内存容器（`isStoredInMemoryOnly: true`，同一份 Schema）。
4. 内存容器构建失败才 `fatalError`（带上下文，而非 `try!` 静默崩溃）。

**关键点**：

- 本地与 iCloud 是**两套独立数据容器**，切换后显示对应容器的数据，**原容器数据不丢失、不自动迁移**。
- iCloud 模式必须保留附件/扫描图的 **inline 字节**（`StorageBackend.requiresInlineImageBytes`），否则 CloudKit 只同步文件名、对端拿不到图。本地模式则把 inline 字节清空、优先读盘。
- 切换 backend 时用 `.id(storageCoordinator.backend)` 强制重建视图树，保证 `@Query` 重新绑定新容器；每个 backend 各播种一次默认成员。

### 3.3 默认种子数据

`seedDefaultMembersIfNeeded` 在每个 backend 各播一份默认家人：**"我 / 爸爸 / 妈妈**"（注：不是 README 早期描述的"爸爸/妈妈/女儿"，以代码为准）。幂等标志按 backend 隔离存 UserDefaults，且 `context.save()` 成功才置位（失败可下次启动重试）。

## 四、端侧 AI 架构（防幻觉为核心）

这是本项目最具技术含量、也最能体现"设计思路"的部分。

### 4.1 Provider 抽象

```
MedicalAIService（单例门面）
        │ currentClient() 取本地强引用（保活旧 client，防 EXC_BAD_ACCESS）
        ▼
   MedicalAIClient 协议
   ├── RuleBasedMedicalAIClient   ← 默认/兜底：Vision OCR + 正则规则
   └── Gemma4EdgeClient           ← 端侧 Gemma，转发给 GemmaEngine
```

- **`rulesOnly`**（默认）：无 LLM，靠 Vision OCR + `MedicineOCRParser` 正则 + 模板化查询回答。App 永不因模型缺失而不可用。
- **`gemma4E2B`**：端侧 Gemma 4 E2B（`.litertlm`，单文件约 2.58GB），默认 GPU 后端。模型就绪后自动切换。

对外统一接口（无论哪个 provider）：

- `extractMedicine(fromOCRText:)` — 从 OCR 文本抽药品
- `visionExtractMedicine(images:hintOCRText:)` — VLM 直接看图抽药
- `analyzeAttachment(ocrText:attachmentType:)` — 附件摘要 + 病历/处方结构化
- `answer(query:scope:)` / `answerStreaming(query:scope:)` — 非流式 / 流式本地问答

**任何 provider 推理抛错都自动降级到规则回退**，保证业务连续。

### 4.2 防幻觉三层设计（端到端）

#### 第一层：提示词硬规则（`MedicalAIPrompting.swift`）

所有任务的 system prompt 都注入 **7 条全局硬规则**：

1. 只能依据用户提供的 OCR 文本、本地结构化记录和明确给出的上下文作答。
2. 禁止根据常识、医学经验或猜测补全缺失事实。
3. 没有证据就回答**"没有"**。
4. 无法从现有上下文确定就回答**"不知道"**。
5. OCR 文本冲突、残缺或歧义时，必须明确指出"信息不确定，需要用户核对原图"。
6. **只输出医疗记录整理结果，不输出诊断建议、治疗建议或用药建议。**
7. 回答必须尽量引用来源字段、来源记录或 OCR 片段。

并辅以任务专项约束，例如：

- 药品抽取："OCR 分组标签（药盒正面/反面/说明书）只是照片类型，**不得作为药品名称**"（修过 `name="药盒正面"` 的 bug）。
- VLM 看图："只根据图片上能直接读到的文字提取……看不清或没有的字段填 null，不要猜，不要编造图片上不存在的信息"。
- 问答："answer 文本里不要写 `[Medicine xxx]` 这种引用标记，引用一律进 citations；涉及用药/诊断决策时提醒用户咨询医生"。

#### 第二层：输出校验与归一化（`GemmaEngine.swift` sanitize 链）

- 所有 payload 字段**全部 Optional**，`"没有"/"不知道"/"null"` 归一化为 `nil`（防模型把字面值塞进 name）。
- `warnings` 为空且 `name == nil` 时**强制塞"无法确认是药品信息"**（防漏报）。
- summary 多级兜底（模型摘要 → 字段拼接 → 兜底文案）。
- 问答文本经 `stripInlineCitations` 正则去掉残留的 `[Medicine <UUID>]` 等标记，避免长 UUID 暴露给用户。

#### 第三层：问答安全门（最严格）

`sanitizeQueryPayload`：**citations 为空时，必须是 `grounded == true` 且答案以"没有/不知道"开头，否则直接 `throw unsafeResponse`**。这一行强制模型"要么有引用、要么老实承认无知"，是防幻觉的最后一道闸。

### 4.3 端侧推理链路（`GemmaEngine` actor）

```
UI: MedicalAIService.shared.answerStreaming(query, scope)
   ↓（取强引用，绕开 actor 让取消立刻生效）
actor GemmaEngine → runSerialized（Task 链排队，单 permit，同一时刻只跑一个推理）
   ↓ loadModelIfNeeded + bridge.resetConversation（每次推理重置，无多轮累积）
performGenerateJSON → generateRaw → tokenStream
   ↓ bridge.sendText(tokenHandler:, completion:)   ← LiteRT-LM 内部线程
JSON 子串提取 → decode payload → sanitize → 业务模型
```

**关键工程要点**：

- **`actor` + `runSerialized`**：曾经踩坑——actor 在 await 时让出隔离，第二次推理进来会 `resetConversation()` 把第一次的会话清掉。修复为 Task 链串行。
- **双层取消**：UI `onTermination` 同时 `task.cancel()` + `MedicalAIService.cancelInflight()` → `GemmaEngine.cancel()`（`nonisolated`，绕开 actor 排队）→ `bridge.cancel()`（原子标志，可跨线程）。曾踩过"点取消要等几秒到下一个 token 边界才停"。
- **流式 token 缓冲**：查询页用 `bufferedTokens`（≥80 字符或 ≥0.05s flush）平滑写入 `streamedAnswer`，避免逐字刷新卡顿。
- **Native bridge 生命周期红线**：绝不在 inference 中释放 bridge（tokenHandler 闭包仍持有 self，会 EXC_BAD_ACCESS）。`MedicalAIService.reload()` 用 stateLock 保活旧 client 直到 in-flight 推理自然结束。

### 4.4 风险确认状态机（`MedicalAIUsageState`）

- `@MainActor` 单例 + UserDefaults 持久化，固定文案：

  > AI 模型可能出现幻觉、遗漏或误读。以下结果仅基于本地记录与 OCR 文本生成，使用前请自行核对原始病历、处方或药盒内容。

- 在**每个 AI 入口**（查询页、药品 AI 抽取、附件智能填充、附件自动摘要）校验：未确认时按钮 `disabled` 或弹"使用前请确认"alert。
- 确认是**一次性 + 可重置**：一旦 `acknowledgeRisk()` 全 App 放行；用户可在设置页 `resetAcknowledgement` 重新激活。

## 五、核心功能与交互逻辑

### 5.1 首页 Today（`HomeView`）

从上到下四个区块，数据全部来自本地 SwiftData `@Query`：

1. **Hero（日期问候）**：中英双行日期（`M 月 d 日 · EEEE` / `MMM d, EEE`）+ 按时段问候（深夜好/早上好/午安/下午好/晚上好），衬线大标题。
2. **今日服药待办**：`TodayScheduleProvider.todayItems(...)` 从 `MedicineItem` 的 `scheduledTimes` + 计划起止日 + 当日 `MedicationLog` 派生；每行 `TodoCheckRow` 可勾选（写 `MedicationLog`），副标题显示剂量 + 低库存提示。无安排时引导"前往药箱"。
3. **复诊倒计时**：取最近未过期的 `followUpDate`，显示"今天"/"还有 N 天"，点击直达病历。
4. **低库存**：`stock <= 5` 的药品列表。
5. **家人快捷入口**：圆形色点头像（`MemberDot`），点击跳病历 Tab。

### 5.2 病历档案

#### 列表页
- 就诊日期倒序 + 按年份分组 + 顶部成员 chip 筛选。
- 通知/首页跳转通过 `AppNavigationState.pendingRecordID` 触发 push 到详情。
- **删除病历的精心设计**：先 fetch 全局 `MedicineItem`，把命中 `sourcePrescriptionID` 的引用置 nil（**保留药箱条目本体**）→ 删附件磁盘文件 + 实体 → 删处方 + 病历 → 清复诊通知。

#### 详情页
- 卡片：基本信息 / 诊断信息 / 处方药品（每条带"已在药箱"/"添加到药箱"按钮）/ 附件照片网格（OCR 全文检索）/ 复诊提醒 / 导出 PDF。
- "添加到药箱"：单条处方 → `MedicineItem`，带完整溯源字段，`stock = 0`。

#### 编辑页（最复杂的页面之一）
- **字段级 OCR**：每个文本框旁的 📷 按钮单独拍照识别回填。
- **整页智能填充**：拍照/选图 → Vision OCR → `analyzeAttachment` → **只填充当前为空的字段**（不覆盖用户已填），处方按 ID diff 增删。
- **处方 → 药箱同步**（`syncPrescriptionsToMedicineBox`，解耦设计的核心）：
  - 已锚定（`sourcePrescriptionID` 命中）→ upsert 信息，**不动 stock/计划**，`stockToAdd > 0` 才累加。
  - 未锚定 → 新建 `MedicineItem`，`sourcePrescriptionID = draft.id`。
  - 取消同步 → 只把 `sourcePrescriptionID = nil`，保留药箱条目。
  - 自动估算入库：`estimatedStock = max(0, 每日剂数 × 疗程天数)`，用户手动改过则不重算。
- **保存流水线**：校验 → 新建/更新病历 → 处方/附件按 ID diff → 同步药箱 → `save()` → 更新复诊通知。

#### 附件查看页
- 首次打开若无 AI 摘要，自动 `analyzeAttachment` 写回 `aiSummary/aiStructuredJSON/aiSearchableText/aiProviderRaw`。
- 展示原图 + AI 识别提示卡 + AI 摘要 + 结构化识别（病历字段 + 处方）+ OCR 文本。

#### PDF 导出
- A4（595×842 点）`UIGraphicsPDFRenderer`，逐行分页，内容：标题 → 基本信息 → 诊断与医嘱 → 处方 → 可选附件图。
- 文件名 `病历-{成员名}-{yyyyMMdd}.pdf`，经 `UIActivityViewController` 分享。

#### 复诊提醒
- 设复诊日期后，`NotificationManager.updateReminder` 排**前一天 9:00 + 当天 9:00** 两条 `UNCalendarNotificationTrigger`。
- `userInfo["recordID"]` → `AppDelegate` 解析 → `AppNavigationState.pendingRecordID` → 列表页消费 → push 详情（通知 deep link）。

### 5.3 家庭药箱

#### 列表页
- 卡片显示：药名、库存（≤5 warning）、剂量·频率、AI 摘要、图片来源（手动 vs 来源医院+日期）。
- **删除药品的手动级联**：删 scan 文件 + 实体 → fetch 裸 UUID 关联的 `MedicationLog` 全删 → 删 `MedicineItem`。

#### 编辑页（药箱侧最复杂页面）
- **关键设计：扫描后只存照片 + OCR，不做自动 AI 提取，必须用户手动点按钮**（避免自动写入错误字段）。
- **智能整理两条通道**（仅模型就绪时显示）：
  - 「AI 智能整理 OCR 文本」→ `extractMedicine(fromOCRText:)`（LLM 通道，快，依赖 OCR 质量）。
  - 「AI 看图提取（实验性）」→ `visionExtractMedicine(images:)`（VLM 通道，直接读图更准但慢，首次加载 5–15s）。
- **扫描采集**：每种 `MedicineScanType`（正/反/侧/说明书/其他）可拍照或选相册，`MediaPermission` 先授权检查；OCR 前经 `ImagePreprocessor` 压缩（长边 ≤1800、JPEG 0.82）。
- **合并 OCR 文本**：`combinedScanOCRText` 拼接所有 scan 的 OCR，**不混入 scanType 中文名**（修过 `name="药盒正面"` bug）。
- **服药计划**：`isActive` 开关 + `scheduledTimes[]`（HH:mm）+ 起止日期。首页今日待办据此派生。

### 5.4 本地查询 / 问答（`MedicalAIQueryView`）

查询页是端侧 AI 的集大成入口：

1. **风险确认门**：展示 `hallucinationDisclosure` + 范围声明，未确认时禁用查询。
2. **检索式召回**（`makeQueryScope`）—— 不依赖向量库，本地数据量小：
   - 清洗问题 → 分词 + 中文 2-gram → 取前 36 个 token。
   - 对病历（最多 8 条）/ 药品（最多 8 条）打分排序（整体命中 +24，token 命中按长度加分）。
   - 拼装每条记录/药品的 `searchableText`（截断控制：附件 1200 字、病历 1800 字、药品 900 字）。
3. **可选联网**（Toggle 开启）：`DuckDuckGoSearch.search(query)` 只发用户问题，5s 超时，最多 5 条摘要作为外部上下文。失败降级不阻断。
4. **流式推理**：`MedicalAIService.answerStreaming`，token 缓冲写入，`.finished(final)` 写最终答案 + citations。流式失败兜底走非流式 `answer`。
5. **结果展示**：答案文本 + grounded 提示 + citations 列表（可跳转）。流式期显示闪烁光标 + 可展开"思考过程"。
6. **并发控制**：`queryGeneration: UInt64` 递增，快速连击/取消时旧 task 在 await 后比对 generation，不等就跳过任何 state 写回（防"误关闭"）。

### 5.5 家庭成员管理（`FamilyManagementView` / `FamilyMemberEditView`）

- List 展示成员（色点 + 关系 + 病历数），滑动删除 / 点击编辑 / 右上加新增。
- **删除成员的归属处理**（关键）：病历 `member` 关系 SwiftData 自动置 nil；但药品 `sourceMemberName`、日志 `memberName` 是**裸字符串**，必须手动 fetch 清空，否则指向不存在的成员。任一类 fetch 失败 → **回滚不删**（防半成品）。
- **改名传播**：`propagateRename` 把旧名关联的药品/日志全改成新名；fetch 失败回滚成员名。注释承认"同名成员会一起被改"是字符串关联 schema 的固有局限，待改 UUID 引用解决。
- 编辑页：姓名 + 9 种关系 Picker（自动联动 emoji）+ 12 个 emoji 头像网格。

### 5.6 AI 模型管理（`AIModelManagementView`）

- 显示当前模式（provider displayName）、模型状态（颜色化）、模型文件名、`statusMessage`。
- **下载源 Picker**：Hugging Face 官方源 / HF-Mirror 镜像。
- **下载控制**（状态机映射按钮）：开始下载 / 暂停 / 重新下载 / 删除（弹确认）/ 重试 / 删除残留文件；downloading 时显示进度条。
- footer：首次约 2.58GB，建议 Wi-Fi。

### 5.7 设置

- 家庭成员入口、数据存储切换（segmented，带确认 dialog + 差异说明 + 错误展示）、AI 能力入口、AI 使用提示（风险确认状态切换）。

## 六、设计思路与关键决策

### 6.1 三条产品哲学

1. **离线优先、本地可追溯**：敏感医疗数据默认本机；AI 推理在端侧；联网是可选且最小化的。
2. **AI 是"受约束的整理器"，不是"自由推理助手"**：只做摘要/抽取/归纳/检索/问答，不诊断、不治疗建议、不脑补。
3. **结果必须可核对**：永远保留原图/OCR 原文，AI 结果标记"待核对"，回答带引用来源。

### 6.2 关键工程决策

| 决策 | 原因 |
|---|---|
| 病历↔药箱用 `sourcePrescriptionID` 软锚定 | 药箱是用户资产（stock/计划手动维护），病历端只能控制链接，不能覆盖/删除 |
| `MedicationLog` 用裸 UUID/字符串而非关系 | 解耦 + 兼容 CloudKit；代价是删药品/成员/改名要手动传播 |
| 三处显式手动级联清理 | SwiftData 默认 nullify 会留孤儿，必须显式删文件 + 清字符串/UUID 字段 |
| provider 抽象 + 规则回退 | App 永不因模型缺失/出错而不可用；模型可热替换 |
| `actor` 串行推理 + 双层取消 | native bridge 非线程安全；曾出现会话互踩、取消延迟 |
| JSON payload 全 Optional + sanitize 安全门 | 强制模型承认无知，防虚构字段 |
| generation 并发模式 | 快速连击/取消时旧 task 不污染新 state |
| 图片双层存储（盘 + inline） | 本地省空间；iCloud 必须保留 inline 让对端有图；展示层文件缺失回退 inline |
| 配置存 UserDefaults、模型路径动态解析 | iOS sandbox 容器 UUID 升级后会变，绝对路径不可靠 |
| 默认"AI 不自动写" | 扫描只存照片+OCR，必须用户手动触发整理，避免错误字段自动入库 |

### 6.3 鲁棒性设计

- **provider 自愈**：`ModelDownloadManager.bootstrapAtLaunch()` 在冷启动同步校正磁盘真实状态 → 配置（模型在但 provider 没切 / 模型不在但 provider 还指向 gemma），故意不调 reload 以避免 inference 中替换 client 触发 native use-after-free。
- **模型文件校验**：`isValidModel`（size ≥ 2GB）+ staging 落盘（`.staged` 临时文件）+ 二次校验，不通过删除并提示重下。
- **存储三级回退** + **种子数据 save 成功才置位**（失败可重试自愈）。
- **`#Predicate` 限制规避**：不支持 optional-unwrap 多表达式 → fetch 全量再 Swift 端 filter。

## 七、视觉设计系统（画面风格）

### 7.1 设计基调：纸感、米色暖调、衬线标题

整体走**"医疗纸本/病历册"**的视觉隐喻：

- **米色暖纸底**（`paperBg`）+ 白色卡片（`surface`），像翻开的笔记本。
- **宋体（Songti SC）衬线大标题** + PingFang 正文，中文阅读质感沉稳、专业、不冰冷。
- **墨黑（`ink`）主文字**，不用纯黑，配多档透明度（`ink2/3/4`、`hairline`）做层级。
- **陶土砖红**（`primary`）做主色，温润、不刺眼，区别于常见医疗 App 的冷蓝。

设计原型在 `medicine_box_design/` 内提供**两套方案**：

- **Steady（稳）**：低饱和纸暖色，成员色点用淡彩——**当前 App 落地的就是这套**。
- **Bold（醒）**：高饱和插画色块（Indigo/Peach/Sage/Honey）——保留为原型对比，未在生产 UI 启用。

## 八、关键实现细节

### 8.1 OCR（Vision）

- `VNRecognizeTextRequest`：`.accurate` + `usesLanguageCorrection = true` + 中英双语 `["zh-Hans", "en-US"]`。
- `Task.detached(priority: .userInitiated)` 避免阻塞主线程。
- 喂入前经 `ImagePreprocessor`（EXIF 方向矫正 + 等比缩小 + JPEG 压缩）。

### 8.2 规则解析器（`MedicineOCRParser`）

- 4 个预编译正则（dosage / frequency / dayText / number），静态缓存避免重复编译。
- 编译失败 DEBUG `assertionFailure` 打断、Release 返回 nil 优雅退化（不再静默永久 nil）。
- `isScanMetadataLine` 过滤"药盒正面"等扫描元数据行，避免被当药名。

### 8.3 模型下载（`ModelDownloadManager`）

- 文件名 `gemma-4-E2B-it.litertlm`，校验门槛 2GB（UI 文案 2.58GB），**可用空间硬门槛 6GB**。
- 存放 `Application Support/Models/`，`excludeFromBackup`。
- **断点续传**：`URLSessionDownloadTask.cancel` 回调拿 `resumeData`，按下载源隔离存 UserDefaults；`downloadTask(withResumeData:)` 续传。
- URLSession：`allowsConstrainedNetworkAccess = false`（低数据模式不下）、`waitsForConnectivity = true`、4 小时资源超时。
- **staging 落盘**：先写 `.staged` → 清 resumeData → 二次校验 → `replaceItemAt` 到正式路径，避免半成品覆盖。

### 8.4 联网搜索（`DuckDuckGoSearch`）

- DuckDuckGo Instant Answer API，`format=json&no_html=1&skip_disambig=1`，5s 超时，最多 5 条。
- **只发用户原始问题**，不传任何病历/药品数据，不持久化问题，开关关闭无任何外部请求。

### 8.5 今日计划派生（`TodayScheduleProvider`）

- 遍历 `medicines where isActive && scheduledTimes 非空 && 在计划区间内`，结合当日 `MedicationLog` 计算 `taken`。
- 排序：未服在前 → 按时间升序。低库存阈值统一为 5。

### 8.6 图片存储与缓存

- `StoredImageFileStore`：`{uuid}-original.jpg` + `{uuid}-thumb.jpg`（maxPixelSize 360, q 0.78）落 `Application Support/MedicineBoxImages/`。
- `CachedStoredImage`：NSCache（countLimit 180 / 64MB）+ ImageIO 下采样（`CGImageSourceCreateThumbnailAtIndex`），优先读盘、文件缺失回退 inline 字节（修 iCloud 对端空白图）。

### 8.7 截图模式

- 启动参数 `-screenshotMode` + `-screenshotTab <tab>`：跳过权限请求、可指定起始 Tab，用于自动化截图。

## 九、技术约束与已知边界

- **无独立测试 target**。
- **单文件大文件**：`MedicineBoxNotesApp.swift` 仍承担模型定义、根导航、病历/药箱/查询/设置主页面及 OCR/PDF/通知服务，持续拆分中。
- **CloudKit 兼容约束**：全模型不用 `.unique`、关系默认 nullify，导致删改需手动级联清理。
- **裸字符串/UUID 关联的局限**：删成员/改名需手动传播，同名成员改名会一起被改（待改 UUID 引用）。
- **存储切换不迁移数据**：本地与 iCloud 是两套容器，切换只切换视图、不搬迁。
- **Gemma 真实推理依赖**：本地 `.litertlm` 模型 + 打包的 native xcframework；首次加载有预热耗时；仅 arm64（真机 + Apple Silicon 模拟器）slice。
- **联网风险**：DuckDuckGo 开关打开会把用户问题发给 DuckDuckGo（不传本地数据）。
- **AI/OCR 结果可能不完整**：页面始终保留原图/OCR 原文供核对。
- **强制浅色模式**（`.preferredColorScheme(.light)`）。
